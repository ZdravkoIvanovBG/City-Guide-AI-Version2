Add a new first tab to the Generated Plan page called "Getting There".
It sits before the Itinerary tab in the pill nav:

Getting There | Itinerary | Hotels | Restaurants | Bonus Activities | Before You Go

---

## Overview

This tab helps the user figure out how to travel from their origin
city to the destination city. The user selects where they are
travelling from, and the tab shows all realistic transport options
(flight, train, bus, ferry) with estimated duration, estimated price
range, and a deep link to the relevant booking platform.

All transport data is generated by Gemini. Nothing is fetched from
a live transport API. Every option includes a clearly visible label:
"Estimated — click to check live prices" so the user understands
these are AI-generated estimates, not real-time quotes.

---

## Origin city input

At the top of the tab, show an origin city selector:

"Travelling from:" [ New York, United States ▾ ]

- Default value: auto-populated from the user's profile if they
  have a "home city" set (add a "Home city" field to the profile
  edit form if it doesn't exist)
- If no home city is set, default to empty with placeholder
  "Where are you travelling from?"
- The input uses the same Gemini-powered autocomplete as the main
  city search on the home page — debounced 400ms, returns 6
  suggestions as the user types
- The destination city is fixed and shown as read-only next to an
  arrow: "New York → Barcelona"
- A "Search routes" button triggers the Gemini call
- If origin equals destination, show a friendly error:
  "You're already there!"

## Gemini prompt for transport options

When the user hits "Search routes", send this to Gemini:

"The user wants to travel from {originCity}, {originCountry} to
{destinationCity}, {destinationCountry}. Their travel dates are
{startDate} to {endDate}.

Return a JSON array of all realistic transport options between
these two cities. For each option include:

{
  'mode': 'flight' | 'train' | 'bus' | 'ferry' | 'drive',
  'available': true | false,
  'summary': 'Direct flight, ~2h 10min',
  'duration': '2h 10min',
  'frequency': 'Multiple daily departures',
  'priceRange': {
    'low': 89,
    'high': 320,
    'currency': 'EUR',
    'note': 'Prices vary by airline and booking time'
  },
  'operators': ['Ryanair', 'Vueling', 'EasyJet'],
  'route': 'JFK / EWR → BCN',
  'stops': 'Direct or 1 stop via Madrid',
  'tips': [
    'Book 6–8 weeks in advance for best prices',
    'Check both JFK and Newark airports for cheaper options'
  ],
  'bookingLinks': [
    {
      'platform': 'Google Flights',
      'url': 'https://www.google.com/travel/flights?q=flights+from+{originCity}+to+{destinationCity}',
      'label': 'Search on Google Flights'
    },
    {
      'platform': 'Skyscanner',
      'url': 'https://www.skyscanner.com/transport/flights/{originCode}/{destCode}/{date}/',
      'label': 'Search on Skyscanner'
    }
  ]
}

Rules:
- Only include modes where available: true if the route is
  genuinely possible by that method. Do not invent unrealistic
  options (e.g. no ferry between New York and Barcelona)

- IMPORTANT — handle cities without direct transport hubs:
  If the origin city does not have its own airport, train station,
  or bus terminal that serves the required route, automatically
  factor in the journey to the nearest relevant hub city.
  For example:
  - Origin is a small town in Bulgaria with no airport → instruct
    the user to travel to Sofia first (nearest major airport),
    then fly from there
  - Origin is a small coastal town in Italy with no train → instruct
    the user to reach the nearest intercity train station first
  In these cases, the summary and route fields must reflect the
  full journey including the first leg:
  e.g. summary: "Travel to Sofia (2h by bus), then direct flight
  to Barcelona (~3h)"
  e.g. route: "Your town → Sofia (bus/car) → BCN"
  The priceRange should include an estimate for both legs combined
  where possible, or note them separately
  The tips array should include a specific tip about getting to
  the hub city:
  e.g. "From [small city], take a bus or taxi to Sofia Central
  Bus Station (~2 hours). FlixBus and several local operators
  run this route regularly."

- For flights: always include if any airport serves either city
  or is within reasonable reach (under ~3 hours travel)
- For trains: include if a direct or connecting rail route exists
  between the two cities or countries
- For buses: include only if long-distance coach services
  realistically operate this route
- For ferry: include only if a genuine ferry route exists
- For drive: include if the cities are on the same landmass and
  within roughly 1500km — include approximate drive time and
  distance. If driving requires crossing through multiple
  countries, mention border crossing notes and any toll roads.
- Price ranges should reflect realistic market rates
- operators should list real airlines / train operators / bus
  companies that serve this route
- If the origin is a small or lesser-known city, explicitly
  acknowledge this in the summary and give clear instructions
  for reaching the nearest transport hub — never silently assume
  the user knows how to get there

Return ONLY the JSON array. No markdown, no preamble."

## Frontend UI

### Loading state
While Gemini generates, show a subtle skeleton loader in the tab —
not a full page takeover, just the tab content area showing
placeholder cards with a shimmer animation.

### Transport option cards

Each available transport mode gets its own card. Stack them
vertically, ordered by: flight first, then train, bus, ferry, drive.

Card layout:

┌─────────────────────────────────────────────────────┐
│  ✈ FLIGHT                          ~2h 10min        │
│  New York → Barcelona                               │
│  Multiple daily departures                          │
│                                                     │
│  Estimated price: €89 – €320                        │
│  Operators: Ryanair · Vueling · EasyJet             │
│  Route: JFK / EWR → BCN · Direct or 1 stop          │
│                                                     │
│  💡 Book 6–8 weeks ahead for best prices            │
│  💡 Check both JFK and Newark for cheaper options   │
│                                                     │
│  [ Search Google Flights ↗ ] [ Skyscanner ↗ ]      │
│  [ Rome2rio ↗ ]                                     │
│                                                     │
│  ⓘ Estimated — click to check live prices          │
└─────────────────────────────────────────────────────┘

Design details:
- Mode icon: ✈ flight, 🚂 train, 🚌 bus, ⛴ ferry, 🚗 drive
- Mode label in small caps at the top left, bold
- Duration top right, large and prominent in Cormorant Garamond
- Price range in amber color
- Operators as pill tags
- Tips with lightbulb icon, same style as destination insider tips
- Booking links as solid buttons — primary button for the most
  relevant platform (Google Flights for flights, Omio for trains),
  secondary outlined buttons for the rest
- All booking links open in a new tab
- "Estimated" disclaimer in small muted text at the bottom of
  each card, always visible — never hidden
- If only one transport mode is available (e.g. only flights for
  an intercontinental route), show a friendly note at the top:
  "For this route, flying is the only practical option"

### No results state
If Gemini returns no available modes (shouldn't happen but handle
it), show:
"We couldn't find route options for this journey. Try searching
on Rome2rio for all available connections." with a link to
rome2rio.com/s/{origin}/{destination}

### Caching
Cache the Gemini transport response in the database keyed by
originCity + destinationCity + startDate. If the same route is
searched again, return the cached result without calling Gemini.
Cache expires after 24 hours.

## Profile — home city field

On the profile edit page, add a "Home city" field:
- Same Gemini autocomplete as everywhere else
- Saved to the user's profile in the database as homeCity (string)
  and homeCountry (string)
- When a user opens the "Getting There" tab on any plan, this value
  auto-populates the origin field
- Label it on the profile page: "Home city — used to pre-fill your
  departure city on travel plans"
